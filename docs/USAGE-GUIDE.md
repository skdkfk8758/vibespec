# VibeSpec 사용 가이드

> Spec-Driven Development (SDD) 워크플로우를 Claude Code에서 실행하는 플러그인

---

## 1. 설치

```bash
# Claude Code 마켓플레이스에서 설치
claude plugin install vibespec

# 또는 직접 설치
claude plugin install /path/to/vibespec
```

설치 후 아무 프로젝트에서 Claude Code를 실행하면 자동으로 활성화됩니다.

---

## 2. 초기 셋업

프로젝트에서 처음 사용할 때:

```
/vs-setup
```

이 명령은:
- 프로젝트의 기술 스택을 자동 감지합니다
- SDD 워크플로우를 설명합니다
- DB를 초기화하고 첫 플랜 생성을 안내합니다

```

  ┌─────────────┐
  │  📝 Backlog │  "나중에 이것도 해야해" / "버그 기록해둬"
  │ /vs-backlog │  → 아이디어/버그/작업을 빠르게 기록 & 관리

  └──────┬──────┘

         ▼
  ┌─────────────┐
  │  📋 Plan    │  "요구사항을 태스크로 분해해줘"

         ▼
  ┌─────────────┐     ┌──────────────┐
  │  🌳 Worktree│     │  🛡️ Guard    │
  │ /vs-worktree│────▶│  /vs-guard   │  격리 환경 + 안전 모드
  └──────┬──────┘     └──────────────┘
         │
         ▼
  ┌─────────────┐     ┌──────────────┐     ┌──────────────┐
  │  ⚡ Execute │     │  📌 Next     │     │  🎯 Pick     │
  │  /vs-exec   │ OR  │  /vs-next    │ OR  │  /vs-pick    │
  │ (일괄 실행) │     │ (순차 실행)  │     │ (선택 실행)  │
  └──────┬──────┘     └──────┬───────┘     └──────┬───────┘

         └───────────────────┼─────────────────────┘

  ┌─────────────┐
  │  ✅ Verify  │  "구현이 스펙을 충족하는지 검증"
  │ /verification│ → 테스트/빌드/lint + AC 크로스체크
  └──────┬──────┘
         │
         ▼
  ┌─────────────┐     ┌──────────────┐
  │  💾 Commit  │     │  🔍 QA       │
  │  /vs-commit │────▶│  /vs-qa      │  QA 에이전트 팀 실행
  └──────┬──────┘     └──────────────┘
         │
         ▼
  ┌─────────────┐
  │  🏷️ Release │  "v1.2.0 릴리즈"
  │  /vs-release│  → changelog + tag + GitHub Release
  └──────┬──────┘
         │ 자동 제안
         ▼
```

## 4. 시나리오별 사용법

### 시나리오 0: 아이디어를 모아두고 싶을 때

```
사용자: "나중에 알림 기능도 추가해야겠다"
사용자: "이 버그 기록해둬"
사용자: "GitHub Issues에서 가져와줘"

/vs-backlog
```

**6가지 모드:**

```
┌───────────────────────────────────────────────────────┐
│                    /vs-backlog                         │
├───────────┬───────────┬───────────┬───────────────────┤
│  빠른추가  │ 목록조회  │ 플랜승격  │   즉시실행        │
│  "기록해"  │ "뭐있지" │ "이거하자"│  "바로해"         │
├───────────┴───────────┼───────────┼───────────────────┤
│     대량 작업          │  임포트    │                   │
│  "정리하자"            │ GitHub    │                   │
│                        │ Slack     │                   │
│                        │ 텍스트    │                   │
└────────────────────────┴───────────┴───────────────────┘
```

| 모드 | 설명 | 예시 |
|------|------|------|
| **빠른 추가** | 한줄로 아이디어/버그 기록 | "로그인 페이지 느려" → `bugfix`, `high` 자동 분류 |
| **목록 조회** | 테이블/칸반 뷰 + 오래된 항목 우선순위 승격 제안 | 7일 이상 미처리 → 자동 승격 제안 |
| **플랜 승격** | 백로그 → `/vs-plan`으로 연결 | 항목의 제목/설명이 플랜 초기 컨텍스트로 전달 |
| **즉시 실행** | `simple` 복잡도 항목을 플래닝 없이 바로 처리 | 한 줄 수정, 설정 변경 등 |
| **대량 작업** | 일괄 Drop/태그/우선순위 변경 | 스프린트 시작 전 정리 |
| **임포트** | GitHub Issues, Slack, 텍스트에서 가져오기 | `--dry-run`으로 미리보기 후 선택 등록 |

**자동 분류 예시:**
```
"로그인 버그 급해" → bugfix / critical / simple
"나중에 다크모드"  → idea / low / moderate
"API 리팩토링"     → refactor / medium / complex
```

**워크플로우 연결:**
```
/vs-backlog (기록) → /vs-backlog (정리) → /vs-backlog (승격) → /vs-plan
```

### 시나리오 A: 아이디어만 있을 때

```
사용자: "사용자 알림 기능을 만들고 싶어"

/vs-ideate 사용자 알림 기능
```

**흐름:**
1. 6가지 질문으로 아이디어를 구체화
2. 복잡도 판단 (S/M/L/XL)
3. 기존 코드베이스 자동 탐색
4. SDD 스펙 초안 자동 생성
5. → `/vs-plan`으로 자동 연결

---

### 시나리오 B: 요구사항이 명확할 때

```
사용자: "OAuth2 로그인 기능을 추가해야 해. Google, GitHub 지원"

/vs-plan
```

**흐름:**
1. 요구사항 인터뷰 (이미 명확하면 빠르게 진행)
2. 코드베이스 탐색
3. **보안 키워드 자동 감지** → Security Considerations 제안
4. SDD 스펙 작성 → 사용자 승인
5. 태스크 분해 (15-30분 단위)
6. 플랜 생성

---

### 시나리오 C: 플랜 실행 — 3가지 방식

#### C-1. 빠르게 전부 실행 (소규모 플랜)
```
/vs-exec
```
- 5개 이하 태스크에 적합
- 단일 세션에서 순차 실행
- 중간에 검증 에이전트 디스패치

#### C-2. 태스크 하나씩 고품질 실행
```
/vs-next
```
- 복잡한 플랜에 적합
- 태스크마다 TDD 에이전트 디스패치
- 독립 검증으로 높은 품질 보장

#### C-3. 특정 태스크만 골라서
```
/vs-pick
```
- 우선순위 높은 태스크를 먼저 처리할 때

---

### 시나리오 D: UI 기능 개발

```
/vs-design-init        ← 1. 디자인 시스템 정의 (DESIGN.md 생성)
/vs-plan               ← 2. 플랜 생성 (자동으로 디자인 리뷰 포함)
/vs-next               ← 3. 태스크 실행
/vs-design-review      ← 4. 구현 후 디자인 감사
/vs-browse http://localhost:3000  ← 5. 브라우저로 시각적 확인
```

**자동 연동:**
- `vs-plan`이 UI 관련 플랜 감지 시 DESIGN.md 존재 확인
- `vs-qa`에서 "디자인 검증" 모드 선택 가능
- `qa-acceptance-tester`가 자동으로 디자인 일관성 검사

---

### 시나리오 E: 안전한 작업

```
/vs-guard on ./src     ← careful + freeze 동시 활성화
/vs-worktree           ← 격리된 워크트리에서 작업
/vs-exec               ← 배치 실행 (파괴적 명령 자동 차단)
/vs-merge              ← 작업 완료 후 메인 브랜치에 병합
```

**차단되는 명령 예시:**
```
⚠️ [careful 모드] rm -rf 명령이 감지되었습니다. 비활성화: vs careful off
🔒 [freeze 모드] /lib/utils.ts 는 freeze 경계(./src) 밖입니다.
```

---

### 시나리오 F: 보안이 중요한 기능

```
/vs-plan               ← 보안 키워드 자동 감지 → Security Considerations 제안
/vs-next               ← 태스크 실행
/vs-security           ← OWASP Top 10 보안 감사
/vs-qa                 ← QA 시 보안 에이전트 자동 디스패치
```

**자동 감사 항목:**
- A01 Broken Access Control
- A02 Cryptographic Failures
- A03 Injection (SQL/XSS/Command)
- A07 Authentication Failures
- A09 Security Logging Failures

## 6. 자동 학습 시스템

VibeSpec은 사용할수록 똑똑해집니다:

```
에러 발생 → error-kb에 기록 → self-improve가 패턴 분석
  → 3회 반복 시 .claude/rules/로 규칙 승격
  → 이후 같은 실수 자동 방지
```

### 관련 명령
| 명령 | 설명 |
|------|------|
| `/vs-commit` (fix 커밋) | 자동으로 self-improve 트리거 |
| `vs error-kb search <키워드>` | 과거 에러 패턴 검색 |
| `vs self-improve status` | 학습 규칙 현황 확인 |

## 8. CLI 명령어 (터미널에서 직접)

스킬 외에 `vs` CLI로 직접 데이터를 조회/관리할 수 있습니다:

```bash
# 플랜 관리
vs plan list                    # 모든 플랜 조회
vs plan show <plan_id>          # 플랜 상세 + 태스크 트리
vs plan list --json             # JSON 출력 (스크립트용)

# 태스크 관리
vs task next <plan_id>          # 다음 실행 가능 태스크
vs task update <id> done        # 태스크 상태 변경

# 가드레일
vs careful on                   # 파괴적 명령 차단 활성화
vs careful off                  # 비활성화
vs freeze set ./src             # 편집 범위 제한
vs freeze off                   # 해제
vs guard on ./src               # careful + freeze 동시
vs guard status                 # 현재 상태 확인


# 아이디어 이력
vs ideate list                  # ideation 이력 조회
vs ideate show <id>             # 상세 조회

# QA
vs qa run list <plan_id>        # QA 실행 이력
vs qa finding list <run_id>     # 발견 이슈 목록

# 설정
vs config list                  # 모든 설정 조회
vs config set <key> <value>     # 설정 변경

# 대시보드
vs dashboard                    # 전체 현황
```

## 9. 팁

1. **매 세션 시작 시** `/vs-resume`을 실행하세요. 이전 컨텍스트가 복원됩니다.

2. **큰 기능은 반드시** `/vs-worktree`로 격리하세요. 메인 브랜치가 안전합니다.

3. **UI 작업 전에** `/vs-design-init`으로 DESIGN.md를 만드세요. 이후 자동으로 디자인 일관성이 검증됩니다.

4. **배치 실행 전에** `/vs-guard on <path>`로 안전 모드를 활성화하세요.

5. **fix 커밋을 하면** 자동으로 에러 패턴이 학습됩니다. 같은 실수를 반복하지 않게 됩니다.

6. **`--json` 플래그**를 붙이면 모든 CLI 출력이 JSON 형식이 됩니다. 스크립트 연동에 유용합니다.
