## 2. 플랜 라이프사이클

### S2.1 — 아이디어 구상 (`/vs-ideate`)
| 항목 | 내용 |
|------|------|
| **실행** | `/vs-ideate` → "사용자 알림 시스템 구현" |
| **검증** | - 6가지 핵심 질문(Why/What/Who/How/When/Risk) 구조화<br>- SDD 스펙 초안 자동 생성<br>- `/vs-plan` 연결 제안 |

### S2.2 — 플랜 생성 (`/vs-plan`)
| 항목 | 내용 |
|------|------|
| **실행** | `/vs-plan` → "사용자 알림 시스템: 이메일/푸시/인앱 알림 통합" |
| **검증** | - 스펙 문서 생성 (Overview → Requirements → Data Model → API → Edge Cases → Success Criteria)<br>- 태스크 트리 자동 분해<br>- plan 상태: `draft`<br>- DB에 plan 레코드 생성 확인 |

### S2.3 — 플랜 목록 조회
| 항목 | 내용 |
|------|------|
| **실행** | `vs plan list` |
| **검증** | - 생성된 플랜 표시<br>- 상태, 진행률, 태스크 수 표시 |

### S2.4 — 플랜 상세 조회
| 항목 | 내용 |
|------|------|
| **실행** | `vs plan show <plan-id>` |
| **검증** | - 플랜 제목, 상태, 스펙, 태스크 트리 표시<br>- 진행률 0% 표시 |

### S2.5 — 플랜 활성화
| 항목 | 내용 |
|------|------|
| **실행** | `vs plan update <plan-id> --status active` |
| **검증** | - 상태 `draft` → `active` 전환<br>- events 테이블에 상태 변경 기록<br>- 대시보드에 활성 플랜 표시 |

### S2.6 — 플랜 승인
| 항목 | 내용 |
|------|------|
| **실행** | `vs plan approve <plan-id>` |
| **검증** | - 상태 `active` → `approved` 전환<br>- 이벤트 기록 확인 |

### S2.7 — 플랜 리뷰 (`/vs-review`)
| 항목 | 내용 |
|------|------|
| **실행** | `/vs-review` |
| **검증** | - 플랜/태스크 인터랙티브 선택<br>- 스펙 및 태스크 수정 가능<br>- 변경사항 DB 반영 |

### S2.8 — 플랜 완료
| 항목 | 내용 |
|------|------|
| **실행** | 모든 태스크 done 후 `vs plan complete <plan-id>` |
| **검증** | - 상태 → `completed`<br>- `completed_at` 타임스탬프 기록<br>- plan_metrics 뷰에서 조회 가능 |

### S2.9 — 플랜 아카이브
| 항목 | 내용 |
|------|------|
| **실행** | `vs plan archive <plan-id>` |
| **검증** | - 상태 → `archived`<br>- 대시보드에서 제외<br>- 데이터 보존 확인 |

### S2.10 — 플랜 삭제
| 항목 | 내용 |
|------|------|
| **실행** | `vs plan delete <plan-id>` |
| **검증** | - 플랜 및 관련 태스크 CASCADE 삭제<br>- 이벤트 기록 확인 |

## 3. 태스크 관리

### S3.1 — 다음 태스크 가져오기 (`/vs-next`)
| 항목 | 내용 |
|------|------|
| **전제조건** | 활성 플랜에 todo 태스크 존재 |
| **실행** | `/vs-next` |
| **검증** | - 의존성 순서 고려한 다음 태스크 반환<br>- 태스크 상태 → `in_progress`<br>- 스펙, acceptance criteria 표시 |

### S3.2 — 특정 태스크 선택 (`/vs-pick`)
| 항목 | 내용 |
|------|------|
| **실행** | `/vs-pick` → 태스크 목록에서 선택 |
| **검증** | - 선택한 태스크 상태 → `in_progress`<br>- 이전 in_progress 태스크 처리 확인 |

### S3.3 — 태스크 상태 업데이트
| 항목 | 내용 |
|------|------|
| **실행** | `vs task update <task-id> --status done` |
| **검증** | - 상태 전환 (todo → in_progress → done)<br>- `completed_at` 기록<br>- 플랜 진행률 업데이트<br>- task_metrics 기록 |

### S3.4 — 태스크 블로킹
| 항목 | 내용 |
|------|------|
| **실행** | `vs task block <task-id> --reason "외부 API 대기"` |
| **검증** | - 상태 → `blocked`<br>- block_reason 기록<br>- 대시보드 알림에 blocked 태스크 표시 |

### S3.5 — 태스크 의존성
| 항목 | 내용 |
|------|------|
| **전제조건** | 태스크 B가 태스크 A에 의존 (`depends_on`) |
| **실행** | 태스크 A 미완료 상태에서 `/vs-next` |
| **검증** | - 태스크 B가 반환되지 않음<br>- 태스크 A 완료 후 태스크 B 반환 |

### S3.6 — 계층형 태스크 트리
| 항목 | 내용 |
|------|------|
| **실행** | `vs task show <parent-task-id>` |
| **검증** | - 부모/자식 태스크 계층 구조 표시<br>- depth, sort_order 정확성 |

### S3.7 — 태스크 스킵
| 항목 | 내용 |
|------|------|
| **실행** | `vs task update <task-id> --status skipped` |
| **검증** | - 상태 → `skipped`<br>- 플랜 진행률에서 제외 또는 별도 카운트 |

### S3.8 — 스코프 제한 (allowed_files / forbidden_patterns)
| 항목 | 내용 |
|------|------|
| **전제조건** | 태스크에 `allowed_files`, `forbidden_patterns` 설정 |
| **실행** | 해당 태스크 진행 중 범위 외 파일 수정 시도 |
| **검증** | - scope_violations 기록<br>- 경고 또는 차단 |

### S4.4 — 플랜 검증 (`/vs-plan-verify`)
| 항목 | 내용 |
|------|------|

### S4.5 — 커밋 (`/vs-commit`)
| 항목 | 내용 |
|------|------|
| **전제조건** | 태스크 진행 중 코드 변경 |
| **실행** | `/vs-commit` |
| **검증** | - 변경사항 논리 단위 그룹화<br>- 태스크 ID 커밋 메시지에 포함<br>- conventional commit 형식 |

## 13. 통합 E2E 시나리오

### E2E-1 — 아이디어부터 배포까지 전체 워크플로우
```
/vs-ideate → /vs-plan → /vs-worktree → /vs-next (반복) →
/vs-commit (반복) → /vs-verification → /vs-plan-verify →
/vs-qa → /vs-merge → /vs-release
```
| 단계 | 검증 포인트 |
|------|------------|
| 아이디어 | 6가지 질문 구조화, 스펙 초안 |
| 플랜 생성 | 스펙 → 태스크 분해 |
| 워크트리 | 격리 환경, 의존성 설치 |
| 태스크 실행 | 순차 진행, 상태 추적, 메트릭 기록 |
| 커밋 | 태스크 ID 포함, conventional format |
| 검증 | AC 확인, 테스트 pass |
| QA | 시나리오 생성, 이슈 감지 |
| 머지 | squash-merge, 리포트 생성 |
| 배포 | 사전검증 → 배포 → 카나리 |
| 릴리즈 | 체인지로그, 태그, GH Release |

### E2E-2 — 백로그에서 플랜 생성 → 실행
```
/vs-backlog add → /vs-backlog (승격) → /vs-plan → /vs-exec → /vs-plan-verify
```

### E2E-3 — 에러 발생 → 자동 학습 → 재발 방지
```
구현 중 에러 발생 → error-kb-suggest 훅 → 해결 → fix 커밋 →
self-improve-trigger 훅 → Error KB 자동 등록 → 규칙 승격 →
다음 세션에서 자동 적용
```

### E2E-4 — QA 파이프라인 전체 사이클
```
/vs-qa → /vs-qa-status → /vs-qa-findings →
이슈 수정 → /vs-qa (재실행) → 모든 시나리오 PASS
```

### E2E-5 — 멀티 플랜 병렬 관리
```
플랜 A 생성 (active) → 플랜 B 생성 (active) →
대시보드에서 두 플랜 모두 표시 →
플랜 A 태스크 진행 → 플랜 B로 전환 →
각 플랜 독립적 진행 확인
```

### E2E-6 — 세션 복원 & 연속 작업
```
세션 1: /vs-plan → /vs-next → 태스크 3개 완료 → 세션 종료
세션 2: /vs-resume → 컨텍스트 복원 → /vs-next → 이어서 작업
```

### E2E-7 — 가드레일 + 워크트리 격리 작업
```
/vs-worktree → /vs-guard → /vs-next →
(범위 외 수정 시도 → 차단) →
(파괴적 명령 시도 → 차단) →
정상 범위 내 작업 완료
```
