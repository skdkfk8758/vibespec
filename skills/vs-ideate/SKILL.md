# vs-ideate (아이디어 검증)

막연한 아이디어를 구조화된 요구사항으로 변환하고, SDD 스펙 초안을 자동 생성합니다.

<HARD-GATE>
스펙이 사용자에게 제시되고 승인되기 전까지, 코드 작성, 프로젝트 스캐폴딩, 구현 액션을 절대 수행하지 마세요.
이 규칙은 프로젝트의 복잡도와 무관하게 항상 적용됩니다.

"이건 너무 간단해서 스펙이 필요 없다"는 판단은 금지합니다.
간단한 프로젝트일수록 검증되지 않은 가정이 가장 많은 낭비를 만듭니다.
스펙은 짧아도 되지만(몇 문장이라도), 반드시 작성하고 승인받아야 합니다.

단순 버그 수정, 오타 교정, 설정 변경은 이 게이트의 적용 대상이 아닙니다 — 직접 처리하세요.
</HARD-GATE>

## Ultrathink Mode

`$ARGUMENTS`에 `--ultrathink`가 포함되면 **고강도 분석 모드**를 활성화합니다.
일반 모드와의 차이점:

| 항목 | 일반 모드 | Ultrathink 모드 |
|------|----------|----------------|
| Phase 0 MCQ | 최대 3개 차원 | **전체 5개 차원** |
| Phase 1 Follow-up 한도 | 질문당 2회 | 질문당 **3회** |
| Phase 1 짧은 답변 기준 | < 20자 | < **40자** |
| Phase 1 추가 질문 | — | **보조 질문 3개** 추가 (아래 참조) |
| Phase 2 분석 | 등급 판정 | 등급 + **리스크 매트릭스** |
| Phase 3 탐색 | 키워드 Grep/Glob | 키워드 + **의존성 그래프 분석** |
| Phase 4 스펙 | 기본 6섹션 | 6섹션 + **Alternatives Considered** + **Migration/Rollback** |
| Phase 5 제시 전 | 바로 제시 | **자기 비판(Self-Critique)** 후 제시 |

### Ultrathink 보조 질문 (Phase 1 확장)

일반 6개 질문 완료 후, 추가로 아래 3개를 진행합니다:

| # | 질문 | 목적 | 스펙 매핑 |
|---|------|------|-----------|
| 7 | **이 기능이 실패하면 어떤 일이 벌어지는가?** | 장애 시나리오 식별 | `edge_cases.failure_modes` |
| 8 | **6개월 후 이 기능은 어떤 모습이어야 하는가?** | 장기 비전 확인 | `requirements.could` + `overview` 보강 |
| 9 | **이 기능과 충돌하거나 영향받는 기존 기능은?** | 의존성/사이드이펙트 파악 | `edge_cases.dependencies` |

### Ultrathink Self-Critique (Phase 5 확장)

스펙을 사용자에게 제시하기 **직전에**, 아래 관점으로 자기 비판을 수행하고 결과를 스펙 말미에 첨부합니다:

```markdown
## Self-Critique (Ultrathink)

### 빠진 것은?
- {스펙에서 누락되었을 수 있는 요구사항이나 엣지 케이스}

### 가정은 무엇인가?
- {명시적으로 검증되지 않은 가정 목록}

### 가장 약한 부분은?
- {스펙에서 가장 불확실하거나 리스크가 높은 섹션}
```

사용자가 Self-Critique 항목을 확인하여 스펙 보완 여부를 결정합니다.

## When to Use

**사용하세요:**
- 아이디어는 있지만 구체적 요구사항이 아직 정리되지 않았을 때
- 새로운 기능을 만들기 전에 범위와 방향을 확인하고 싶을 때
- 여러 아이디어 중 어떤 것을 먼저 할지 판단이 필요할 때

**사용하지 마세요:**
- 이미 요구사항이 명확한 경우 → `/vs-plan`으로 직접 진행
- 단순 버그 수정이나 리팩토링 → 직접 구현
- 기존 플랜의 태스크를 실행하려는 경우 → `/vs-next`

## Steps

### Phase 0: 요구사항 정제 (Clarify)

모호한 입력을 구조화된 MCQ로 정밀화합니다. clarify 패턴(plugins-for-claude-natives)에서 영감.

#### 스킵 조건 (아래 중 하나라도 충족하면 Phase 0을 건너뛰세요)
- `$ARGUMENTS`에 `--skip-clarify` 포함
- 사용자 입력이 **100자 이상**이고, 아래 핵심 차원 중 **2개 이상** 키워드가 존재

  | 차원 | 감지 키워드 (하나라도 있으면 해당 차원 "명확") |
  |------|----------------------------------------------|
  | scope | "기능", "페이지", "API", "화면", "모듈", "서비스", "엔드포인트" |
  | user | "사용자", "관리자", "개발자", "고객", "팀", "유저", "대상" |
  | tech | "React", "Node", "Python", "TypeScript", "DB", "API", "REST", "GraphQL", 또는 프레임워크/라이브러리명 |

- 스킵 시: "Phase 0 스킵 — 입력이 충분히 상세합니다" 표시 후 Phase 1로 직행

#### 모호성 탐지 + MCQ 생성

사용자 입력(`$ARGUMENTS` 또는 첫 메시지)에서 모호한 차원을 식별하세요.

**5가지 차원과 MCQ 템플릿**:

| 우선순위 | 차원 | 모호 판단 | MCQ 질문 | 선택지 예시 |
|---------|------|----------|----------|------------|
| 1 | **scope** | 기능 범위 키워드 없음 | "이 기능의 범위는?" | "핵심 기능만 (MVP)", "핵심 + 부가 기능", "전체 시스템", "아직 미정" |
| 2 | **user** | 대상 사용자 키워드 없음 | "주요 사용자는?" | "최종 사용자 (일반인)", "개발자/기술 팀", "관리자/운영 팀", "아직 미정" |
| 3 | **tech** | 기술 관련 키워드 없음 | "기술 스택 선호는?" | "기존 프로젝트 스택 유지", "특정 기술 지정 있음", "추천받고 싶음", "아직 미정" |
| 4 | **priority** | MUST/SHOULD 구분 없음 | "우선순위는?" | "급함 (이번 주)", "중요하지만 여유 있음", "아이디어 단계", "백로그용" |
| 5 | **timeline** | 시간 제약 미언급 | "목표 일정은?" | "MVP 먼저 빠르게", "완성도 높게 한번에", "단계적 릴리즈", "일정 미정" |

**실행 규칙**:
1. 모호한 차원을 우선순위 순으로 정렬
2. 상위 **최대 3개**만 선택 (사용자 피로 방지) — `--ultrathink` 시 **전체 5개**
3. `AskUserQuestion`으로 한번에 제시 (questions 배열에 최대 3개):
   - 각 질문의 header: 차원명 (예: "범위", "사용자", "기술")
   - multiSelect: false
   - 사용자가 "Other"를 선택한 경우 (EC2): 자유 입력 텍스트를 해당 차원의 답변으로 사용. MCQ 선택지에 맞지 않는 답변도 정상 처리
4. 모호한 차원이 0개면 Phase 0 스킵

#### before/after 비교 표시

MCQ 답변 수집 후, 아래 형태로 출력하세요:

```
## 요구사항 정제 결과

| 항목 | Before (원본) | After (정제) |
|------|---------------|--------------|
| 범위 | {원본 텍스트 또는 "미정의"} | {선택된 답변} |
| 사용자 | {원본 텍스트 또는 "미정의"} | {선택된 답변} |
| ... | ... | ... |
```

#### Phase 0 → Phase 1 시드 연결

정제 결과를 Phase 1의 6개 질문에 매핑하여 시드 데이터를 구성하세요:

| Phase 0 차원 | Phase 1 질문 매핑 | 시드 활용 |
|-------------|------------------|----------|
| scope | Q5 (MVP) | "범위가 {scope 답변}로 확인됨" → Q5 답변 시드 |
| user | Q1 (사용자) | "대상 사용자가 {user 답변}로 확인됨" → Q1 답변 시드 |
| tech | Phase 3 (코드베이스 탐색) | 기술 키워드를 검색어에 추가 |
| priority | Q5 (MVP) 보충 | 우선순위 정보를 MVP 판단에 반영 |
| timeline | Q5 (MVP) 보충 | 일정 정보를 MVP 범위 결정에 반영 |

**시드된 질문 처리**: Phase 1에서 시드가 있는 질문은 확인만 수행:
- "대상 사용자가 '{user 답변}'인 것으로 확인되었습니다. 맞으면 넘어가겠습니다."
- 사용자가 수정을 원하면 기존 adaptive follow-up으로 진행

### Phase 1: 문제 정의 (6 Forcing Questions + Adaptive Follow-up)

사용자에게 아이디어를 설명해달라고 요청한 뒤, 아래 6가지 핵심 질문으로 인터뷰를 진행하세요.
**Phase 0에서 시드된 질문은 확인만 수행하고 건너뛰세요.**
한 번에 모든 질문을 던지지 말고, 답변을 소화한 뒤 후속 질문을 이어가세요.
사용자의 초기 설명에서 이미 답변된 항목은 건너뛰세요.

| # | 질문 | 목적 | 스펙 매핑 |
|---|------|------|-----------|
| 1 | **이 문제를 겪는 사용자는 누구인가?** | 대상 사용자 정의 | `overview.whom` |
| 2 | **현재 어떻게 해결하고 있는가?** | 기존 대안 파악 | `edge_cases` |
| 3 | **왜 지금 해결해야 하는가?** | 긴급성과 동기 확인 | `overview.why` |
| 4 | **성공을 어떻게 측정할 수 있는가?** | 성공 지표 정의 | `success_criteria` |
| 5 | **가장 작은 의미 있는 버전(MVP)은?** | 범위 제한 | `requirements.must` |
| 6 | **가장 큰 리스크는 무엇인가?** | 불확실성 식별 | `edge_cases`, `non_functional` |

`$ARGUMENTS`에 아이디어 설명이 있으면 이를 출발점으로 사용하세요.

#### Adaptive Follow-up Rules

각 메인 질문의 답변 후, 답변 길이가 **20자 미만** (`--ultrathink` 시 **40자 미만**)이면 후속 질문으로 구체화하세요:

- **질문당 최대 2회** follow-up (`--ultrathink` 시 **3회**). 소진 시 다음 메인 질문으로 이동
- follow-up도 자유 대화로 자연스럽게 진행 (AskUserQuestion 강제 아님)
- 답변이 기준 이상이면 충분한 것으로 판단하고 다음 메인 질문으로
- `--ultrathink` 시: 6개 메인 질문 완료 후 **보조 질문 3개** (Q7-Q9) 추가 진행 (위 Ultrathink Mode 섹션 참조)

**Follow-up 예시:**

```
Q1: "이 문제를 겪는 사용자는 누구인가?"
A:  "개발자" (6자 → 후속 질문 트리거)
→ Follow-up 1: "시니어/주니어? 프론트엔드/백엔드/풀스택?"
A:  "프론트엔드 시니어" (16자 → 후속 질문 트리거)
→ Follow-up 2: "특정 팀 규모나 환경이 중요한가요?"
A:  "스타트업 5인 이하 팀" → Q2로 이동 (2회 소진)

Q5: "가장 작은 MVP는?"
A:  "로그인이랑 결제" (14자 → 후속 질문 트리거)
→ Follow-up 1: "로그인 방식은? 소셜/이메일/전화번호?"
→ Follow-up 2: "결제는 일회성? 구독? 어떤 PG?"
```

#### 점진적 스펙 반영

각 질문의 답변이 확보되면, 내부적으로 스펙 필드를 즉시 매핑하세요:
- Q1 답변 → `spec.overview.whom` 채움
- Q2 답변 → `spec.edge_cases` (기존 대안 한계) 추가
- Q3 답변 → `spec.overview.why` 채움
- Q4 답변 → `spec.success_criteria` 채움
- Q5 답변 → `spec.requirements.must` 채움
- Q6 답변 → `spec.edge_cases` + `spec.non_functional` 추가

Phase 4에서는 이미 매핑된 필드를 기반으로 **보완만** 수행합니다.

### Phase 2: 스코프 결정

인터뷰 결과를 기반으로 복잡도를 판단하세요:

| 등급 | 기준 | 예상 태스크 |
|------|------|-------------|
| **S** (Small) | 1-2개 파일, 단일 모듈 | 1-3개 |
| **M** (Medium) | 3-5개 파일, 2개 이하 모듈 | 3-5개 |
| **L** (Large) | 5개+ 파일, 여러 모듈 | 5-8개 |
| **XL** (Extra Large) | 여러 모듈 + 신규 인프라 | 플랜 분할 권장 |

- XL인 경우 사용자에게 MVP 기준 분할을 제안하세요
- 복잡도와 함께 예상 영향 범위(파일/모듈)를 요약하세요

**Ultrathink 추가 — 리스크 매트릭스**:
`--ultrathink` 시 스코프 판정과 함께 아래 매트릭스를 출력하세요:

```markdown
### Risk Matrix
| 리스크 | 영향도 (H/M/L) | 발생 확률 (H/M/L) | 대응 전략 |
|--------|---------------|-------------------|----------|
| {Q6 답변 기반} | | | |
| {Q7 답변 기반 — 장애 시나리오} | | | |
| {Q9 답변 기반 — 기존 기능 충돌} | | | |
```

### Phase 3: 코드베이스 탐색

인터뷰에서 파악된 키워드로 기존 코드베이스를 탐색하세요:

1. 사용자 답변에서 핵심 키워드(기능명, 모듈명, API 이름 등)를 추출
2. Grep/Glob으로 관련 파일과 모듈을 탐색
3. 관련 코드의 현재 구조와 패턴을 요약
4. 탐색 결과를 Phase 4의 스펙 초안에 반영

**Ultrathink 추가 — 의존성 그래프 분석**:
`--ultrathink` 시 키워드 탐색에 더해 아래를 수행하세요:
1. 관련 모듈의 import/export 관계를 추적하여 의존성 그래프 요약
2. 변경 시 영향받는 파일 목록을 "blast radius"로 제시
3. Q9 답변(기존 기능 충돌)과 교차 검증

**신규 프로젝트인 경우** (파일이 거의 없으면): 이 단계를 스킵하세요.

### Phase 4: SDD 스펙 초안 완성

Phase 1에서 점진적으로 매핑된 스펙 필드를 기반으로, 빈 필드를 보완하고 전체 스펙을 완성하세요:

```markdown
## Overview
{무엇을}, {누구를 위해}, {왜} — 인터뷰 Q1, Q3 기반

## Requirements
### 기능 요구사항
- MUST: {MVP 핵심 기능 — Q5 기반}
- SHOULD: {추가 기능}
- COULD: {향후 확장}

### 비기능 요구사항
- {성능, 보안, 접근성 등 — Q4, Q6 기반}

## Data Model
{기존 코드 탐색 결과 + 신규 엔티티 제안}

## API / Interface
{예상되는 인터페이스 정의}

## Edge Cases
- {Q6에서 식별된 리스크 → 엣지 케이스로 변환}
- {Q2에서 파악된 기존 대안의 한계점}

## Success Criteria
- {Q4의 성공 지표를 정량적 기준으로 변환}

## Alternatives Considered (--ultrathink only)
- {검토했으나 채택하지 않은 접근 방식과 그 이유}

## Migration & Rollback (--ultrathink only)
- 마이그레이션: {기존 → 신규 전환 절차}
- 롤백: {문제 발생 시 복구 절차}
```

### Phase 5: 사용자 확인 및 vs-plan 연결

1. **스펙 초안을 사용자에게 제시**하세요

2. **체크포인트**: `AskUserQuestion`으로 확인:
   - question: "스펙 초안을 검토해주세요. 어떻게 진행할까요?"
   - header: "스펙 확인"
   - 선택지:
     - label: "vs-plan으로 진행", description: "이 스펙으로 플랜을 생성합니다"
     - label: "스펙 수정", description: "피드백을 반영하여 스펙을 수정합니다"
     - label: "저장만", description: "스펙을 context_log에 저장하고 나중에 활용합니다"

3. **"vs-plan으로 진행"** 선택 시:
   - 스펙을 `vs plan create --title "..." --spec "..."` 명령으로 전달
   - `/vs-plan`의 Phase 2(코드베이스 탐색)부터 이어서 진행

4. **"저장만"** 선택 시:
   - **전체 스펙 초안을 저장**: Bash 도구로 아래 명령을 실행하세요:
     ```bash
     vs ideate save --title "{아이디어 제목}" --summary "{1줄 요약}"
     ```
   - 스펙 초안은 ideation 레코드에 저장되며, vs-plan이 `vs ideate list --json`으로 복원합니다

## Rules

- 인터뷰는 대화형으로, 한 번에 1-2개 질문만
- 이미 명확한 답변이 있는 질문은 건너뛰기
- **답변 < 20자 (ultrathink: 40자) → 후속 질문 트리거, 질문당 최대 2회 (ultrathink: 3회)**
- **각 질문 답변 후 대응하는 스펙 필드를 즉시 내부 매핑**
- **ultrathink: Q7-Q9 보조 질문 진행, Self-Critique 수행 후 제시**
- 스펙 초안은 완벽할 필요 없음 — vs-plan에서 보완될 예정
- XL 복잡도에서는 반드시 분할을 제안
- context_log 저장 시 반드시 `[ideation]` 태그 포함
