# vs-ideate (아이디어 검증)

막연한 아이디어를 구조화된 요구사항으로 변환하고, SDD 스펙 초안을 자동 생성합니다.

<HARD-GATE>
스펙이 사용자에게 제시되고 승인되기 전까지, 코드 작성, 프로젝트 스캐폴딩, 구현 액션을 절대 수행하지 마세요.
이 규칙은 프로젝트의 복잡도와 무관하게 항상 적용됩니다.

"이건 너무 간단해서 스펙이 필요 없다"는 판단은 금지합니다.
간단한 프로젝트일수록 검증되지 않은 가정이 가장 많은 낭비를 만듭니다.
스펙은 짧아도 되지만(몇 문장이라도), 반드시 작성하고 승인받아야 합니다.

단순 버그 수정, 오타 교정, 설정 변경은 이 게이트의 적용 대상이 아닙니다 — 직접 처리하세요.
</HARD-GATE>

## When to Use

**사용하세요:**
- 아이디어는 있지만 구체적 요구사항이 아직 정리되지 않았을 때
- 새로운 기능을 만들기 전에 범위와 방향을 확인하고 싶을 때
- 여러 아이디어 중 어떤 것을 먼저 할지 판단이 필요할 때

**사용하지 마세요:**
- 이미 요구사항이 명확한 경우 → `/vs-plan`으로 직접 진행
- 단순 버그 수정이나 리팩토링 → 직접 구현
- 기존 플랜의 태스크를 실행하려는 경우 → `/vs-next`

## Steps

### Phase 1: 문제 정의 (6 Forcing Questions + Adaptive Follow-up)

사용자에게 아이디어를 설명해달라고 요청한 뒤, 아래 6가지 핵심 질문으로 인터뷰를 진행하세요.
한 번에 모든 질문을 던지지 말고, 답변을 소화한 뒤 후속 질문을 이어가세요.
사용자의 초기 설명에서 이미 답변된 항목은 건너뛰세요.

| # | 질문 | 목적 | 스펙 매핑 |
|---|------|------|-----------|
| 1 | **이 문제를 겪는 사용자는 누구인가?** | 대상 사용자 정의 | `overview.whom` |
| 2 | **현재 어떻게 해결하고 있는가?** | 기존 대안 파악 | `edge_cases` |
| 3 | **왜 지금 해결해야 하는가?** | 긴급성과 동기 확인 | `overview.why` |
| 4 | **성공을 어떻게 측정할 수 있는가?** | 성공 지표 정의 | `success_criteria` |
| 5 | **가장 작은 의미 있는 버전(MVP)은?** | 범위 제한 | `requirements.must` |
| 6 | **가장 큰 리스크는 무엇인가?** | 불확실성 식별 | `edge_cases`, `non_functional` |

`$ARGUMENTS`에 아이디어 설명이 있으면 이를 출발점으로 사용하세요.

#### Adaptive Follow-up Rules

각 메인 질문의 답변 후, 답변 길이가 **20자 미만**이면 후속 질문으로 구체화하세요:

- **질문당 최대 2회** follow-up. 2회 소진 시 다음 메인 질문으로 이동
- follow-up도 자유 대화로 자연스럽게 진행 (AskUserQuestion 강제 아님)
- 답변이 20자 이상이면 충분한 것으로 판단하고 다음 메인 질문으로

**Follow-up 예시:**

```
Q1: "이 문제를 겪는 사용자는 누구인가?"
A:  "개발자" (6자 → 후속 질문 트리거)
→ Follow-up 1: "시니어/주니어? 프론트엔드/백엔드/풀스택?"
A:  "프론트엔드 시니어" (16자 → 후속 질문 트리거)
→ Follow-up 2: "특정 팀 규모나 환경이 중요한가요?"
A:  "스타트업 5인 이하 팀" → Q2로 이동 (2회 소진)

Q5: "가장 작은 MVP는?"
A:  "로그인이랑 결제" (14자 → 후속 질문 트리거)
→ Follow-up 1: "로그인 방식은? 소셜/이메일/전화번호?"
→ Follow-up 2: "결제는 일회성? 구독? 어떤 PG?"
```

#### 점진적 스펙 반영

각 질문의 답변이 확보되면, 내부적으로 스펙 필드를 즉시 매핑하세요:
- Q1 답변 → `spec.overview.whom` 채움
- Q2 답변 → `spec.edge_cases` (기존 대안 한계) 추가
- Q3 답변 → `spec.overview.why` 채움
- Q4 답변 → `spec.success_criteria` 채움
- Q5 답변 → `spec.requirements.must` 채움
- Q6 답변 → `spec.edge_cases` + `spec.non_functional` 추가

Phase 4에서는 이미 매핑된 필드를 기반으로 **보완만** 수행합니다.

### Phase 2: 스코프 결정

인터뷰 결과를 기반으로 복잡도를 판단하세요:

| 등급 | 기준 | 예상 태스크 |
|------|------|-------------|
| **S** (Small) | 1-2개 파일, 단일 모듈 | 1-3개 |
| **M** (Medium) | 3-5개 파일, 2개 이하 모듈 | 3-5개 |
| **L** (Large) | 5개+ 파일, 여러 모듈 | 5-8개 |
| **XL** (Extra Large) | 여러 모듈 + 신규 인프라 | 플랜 분할 권장 |

- XL인 경우 사용자에게 MVP 기준 분할을 제안하세요
- 복잡도와 함께 예상 영향 범위(파일/모듈)를 요약하세요

### Phase 3: 코드베이스 탐색

인터뷰에서 파악된 키워드로 기존 코드베이스를 탐색하세요:

1. 사용자 답변에서 핵심 키워드(기능명, 모듈명, API 이름 등)를 추출
2. Grep/Glob으로 관련 파일과 모듈을 탐색
3. 관련 코드의 현재 구조와 패턴을 요약
4. 탐색 결과를 Phase 4의 스펙 초안에 반영

**신규 프로젝트인 경우** (파일이 거의 없으면): 이 단계를 스킵하세요.

### Phase 4: SDD 스펙 초안 완성

Phase 1에서 점진적으로 매핑된 스펙 필드를 기반으로, 빈 필드를 보완하고 전체 스펙을 완성하세요:

```markdown
## Overview
{무엇을}, {누구를 위해}, {왜} — 인터뷰 Q1, Q3 기반

## Requirements
### 기능 요구사항
- MUST: {MVP 핵심 기능 — Q5 기반}
- SHOULD: {추가 기능}
- COULD: {향후 확장}

### 비기능 요구사항
- {성능, 보안, 접근성 등 — Q4, Q6 기반}

## Data Model
{기존 코드 탐색 결과 + 신규 엔티티 제안}

## API / Interface
{예상되는 인터페이스 정의}

## Edge Cases
- {Q6에서 식별된 리스크 → 엣지 케이스로 변환}
- {Q2에서 파악된 기존 대안의 한계점}

## Success Criteria
- {Q4의 성공 지표를 정량적 기준으로 변환}
```

### Phase 5: 사용자 확인 및 vs-plan 연결

1. **스펙 초안을 사용자에게 제시**하세요

2. **체크포인트**: `AskUserQuestion`으로 확인:
   - question: "스펙 초안을 검토해주세요. 어떻게 진행할까요?"
   - header: "스펙 확인"
   - 선택지:
     - label: "vs-plan으로 진행", description: "이 스펙으로 플랜을 생성합니다"
     - label: "스펙 수정", description: "피드백을 반영하여 스펙을 수정합니다"
     - label: "저장만", description: "스펙을 context_log에 저장하고 나중에 활용합니다"

3. **"vs-plan으로 진행"** 선택 시:
   - 스펙을 `vs plan create --title "..." --spec "..."` 명령으로 전달
   - `/vs-plan`의 Phase 2(코드베이스 탐색)부터 이어서 진행

4. **"저장만"** 선택 시:
   - **전체 스펙 초안을 저장**: Bash 도구로 아래 2개 명령을 실행하세요:
     ```bash
     # 1. 전체 스펙 초안 저장 (나중에 vs-plan이 복원할 수 있도록)
     # 스펙이 길면 heredoc 사용:
     vs context save --summary "$(cat <<'SPEC'
     [ideation-spec] {전체 스펙 초안 텍스트}
     SPEC
     )"
     # 2. 요약 저장 (목록 조회용)
     vs context save --summary "[ideation] {아이디어 제목}: {1줄 요약}"
     ```
   - `[ideation-spec]` 태그에 전체 스펙 초안을 포함하여, vs-plan이 이를 복원할 수 있게 하세요
   - `[ideation]` 태그에는 제목+요약만 (목록 표시용)

## Rules

- 인터뷰는 대화형으로, 한 번에 1-2개 질문만
- 이미 명확한 답변이 있는 질문은 건너뛰기
- **답변 < 20자 → 후속 질문 트리거, 질문당 최대 2회**
- **각 질문 답변 후 대응하는 스펙 필드를 즉시 내부 매핑**
- 스펙 초안은 완벽할 필요 없음 — vs-plan에서 보완될 예정
- XL 복잡도에서는 반드시 분할을 제안
- context_log 저장 시 반드시 `[ideation]` 태그 포함

## 다음 단계

- → `/vs-plan`으로 스펙을 플랜으로 발전
- → `/vs-ideate`를 다시 실행하여 다른 아이디어 탐색
- → `vs ideate list`로 이전 ideation 이력 조회
