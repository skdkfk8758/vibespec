# Task Decomposition

스펙을 실행 가능한 태스크 단위로 분해하는 방법론. 적절한 크기, 올바른 순서, 검증 가능한 완료 기준이 핵심이다.

## Domain Context

**태스크 분해**는 SDD 워크플로우에서 스펙과 구현 사이를 잇는 다리다. 스펙이 "무엇을 만들지"를 정의하면, 태스크 분해는 "어떤 순서로 어떤 단위로 만들지"를 결정한다.

좋은 태스크 분해는 INVEST 원칙(Bill Wake)과 SDD의 검증 가능성 원칙을 결합한다.

### Key Principles

1. **15~30분 단위로 분해한다.** 하나의 AI 세션 또는 짧은 집중 시간 내에 완료 가능한 크기.

2. **의존성 순서를 명확히 한다.** 데이터 모델 → 핵심 로직 → API/인터페이스 → 통합/마무리 순서가 일반적이다.

3. **각 태스크는 독립적으로 검증 가능하다.** 태스크 완료 후 테스트를 실행하거나 동작을 확인할 수 있어야 한다.

4. **플랜은 5개 이하를 목표로 한다.** 5개를 초과하면 플랜 분할을 검토한다.

5. **불확실한 부분을 먼저 한다.** 기술적 위험이 높은 태스크를 앞에 배치한다.

## 의존성 순서 결정

### 일반적인 순서 패턴

1. 데이터 모델 / 스키마
2. 핵심 비즈니스 로직 (모델, 유틸리티)
3. API / 인터페이스 (CLI 명령, 엔드포인트)
4. 통합 / 마무리 (설정, 문서, CI)

### 의존성 판단 질문

- B가 완료되지 않으면 A의 테스트를 작성할 수 없는가? → 의존
- B의 결과물(함수, 타입, 스키마)을 A가 import하는가? → 의존
- A와 B를 동시에 진행해도 충돌이 없는가? → 독립

## Acceptance Criteria 작성법

### 좋은 AC의 조건

1. **검증 가능하다**: 자동 테스트 또는 수동 확인으로 참/거짓 판단 가능
2. **구체적이다**: 기대하는 입력, 출력, 상태 변화를 명시
3. **독립적이다**: 다른 AC와 분리하여 개별 검증 가능
4. **완전하다**: 이 AC들을 모두 통과하면 태스크가 완료된 것

### AC 작성 패턴

Given [사전 조건] When [동작] Then [기대 결과]

**나쁜 예**: "에러를 적절히 처리한다"
**좋은 예**: "plan_id가 존재하지 않을 때 `vp plan show` 호출 시 'Plan not found' 에러를 반환한다"

## INVEST 원칙

| 원칙 | 의미 | 점검 질문 |
|------|------|-----------|
| **I**ndependent | 다른 태스크에 최소한으로 의존 | 이 태스크만 완료해도 검증 가능한가? |
| **N**egotiable | 구현 방법은 유연 | spec이 "무엇"만 정의하고 "어떻게"는 열어두었는가? |
| **V**aluable | 독립적으로 가치 제공 | 이 태스크만 완료해도 뭔가가 동작하는가? |
| **E**stimable | 크기 추정 가능 | 15~30분 내 완료 가능한지 판단할 수 있는가? |
| **S**mall | 한 세션에 완료 가능한 크기 | 30분을 초과할 것 같으면 분할이 필요한가? |
| **T**estable | 완료를 검증 가능 | AC가 테스트로 변환 가능한가? |

## 플랜 크기 판단

| 태스크 수 | 판단 | 액션 |
|-----------|------|------|
| 1~3개 | 적절 | 바로 실행 |
| 4~5개 | 적절 | 바로 실행, 단 의존성 순서 주의 |
| 6~7개 | 경계 | 2개 플랜으로 분할 검토 |
| 8개 이상 | 과대 | 반드시 분할 |
