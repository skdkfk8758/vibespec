# Spec → Plan 생성 워크플로우

사용자의 요구사항을 SDD 스펙으로 정리하고 실행 가능한 플랜으로 분해합니다.

## When to Use

**사용하세요:**
- 3개 이상 파일을 변경하는 기능 개발
- 새로운 API, 데이터 모델, 모듈을 추가할 때
- 설계 결정이 필요한 작업 (트레이드오프, 여러 구현 방식)
- 여러 세션에 걸쳐 진행할 규모의 작업

**사용하지 마세요:**
- 단일 파일 수정, 버그 픽스, 오타 수정 → 직접 수정
- 설정 변경, 의존성 업데이트 → 직접 처리
- 기존 UI의 스타일링만 변경 → 직접 처리
- 이미 스펙이 명확한 단순 태스크 → Bash 도구로 `vs task create --plan <id> --title "..."` 명령을 실행하여 바로 생성

## Steps

0. **코드베이스 선행 탐색 + 자동추론**

   인터뷰 **이전에** 코드베이스를 탐색하여 필수 확인 항목 중 2개를 자동으로 추론합니다.

   a. 사용자의 초기 발화에서 관련 모듈/파일 힌트를 추출하세요 (키워드, 파일명, 모듈명 등)
   b. Glob/Grep/Read 도구로 관련 코드를 **병렬로** 탐색하세요:
      - 언급된 파일/모듈을 읽어 현재 구조와 패턴을 파악
      - 기존 코드의 컨벤션(네이밍, 에러 처리 방식, 테스트 패턴 등)을 확인
      - 사용 중인 기술 스택/라이브러리/패턴을 식별
   c. 탐색 결과에서 **자동추론**하세요:
      - **항목5 (기존 코드 제약)**: 수정 대상 파일, 건드리면 안 되는 영역, 기존 코드 구조
      - **항목6 (기술 스택/방식)**: 사용 중인 프레임워크, 라이브러리, 코딩 패턴
   d. 추론 결과를 내부적으로 보관하여 Step 1 인터뷰에서 활용하세요
   e. **Fallback**: 관련 코드를 찾지 못하면 해당 항목을 `[감지실패]`로 표시하고, Step 1 인터뷰에서 기존처럼 직접 질문하세요

1. **요구사항 인터뷰**

   **Ideation 참조**: 인터뷰 시작 전에 Bash 도구로 `vs ideate list --json`을 실행하여 최근 ideation 기록을 확인하세요.
   - `[ideation]` 태그가 붙은 최근 기록이 있으면:
     1. `vs --json context search "[ideation-spec]"`으로 전체 스펙 초안을 검색하세요 (vs-ideate가 저장한 전체 스펙)
     2. 전체 스펙이 있으면: 해당 스펙을 Step 4의 초안으로 제시하고, Step 1~3을 빠르게 확인만 (이미 답변된 항목 건너뛰기)
     3. 전체 스펙이 없고 요약만 있으면: 요약을 요구사항 초안으로 제시하고, 부족한 항목을 추가 인터뷰
   - 기록이 없으면: 아래 일반 인터뷰 플로우를 진행하세요.

   사용자에게 무엇을 만들고 싶은지 물어본 뒤, 아래 **필수 확인 항목**을 모두 파악할 때까지 인터뷰를 진행하세요.
   한 번에 모든 질문을 쏟아내지 말고, 사용자의 답변을 소화한 뒤 후속 질문을 이어가세요.

   ### 1a. 필수 확인 항목

   아래 항목이 모두 파악되어야 스펙 작성으로 넘어갈 수 있습니다.
   사용자의 초기 설명에서 이미 파악된 항목은 건너뛰세요.

   | # | 항목 | 확인할 내용 | 질문 예시 | 자동감지 |
   |---|------|-------------|-----------|----------|
   | 1 | **목적과 동기** | 왜 이 기능이 필요한지, 어떤 문제를 해결하는지 | "이 기능이 필요하게 된 배경이 뭔가요?" | - |
   | 2 | **대상 사용자** | 누가 이 기능을 쓰는지 (최종 사용자, 개발자, 내부 시스템 등) | "이 기능의 주요 사용자는 누구인가요?" | - |
   | 3 | **핵심 유즈케이스** | 사용자가 어떤 흐름으로 기능을 사용하는지 (happy path) | "사용자가 이 기능을 쓰는 가장 일반적인 시나리오를 설명해주세요" | - |
   | 4 | **입출력 경계** | 기능의 입력은 무엇이고, 기대하는 출력/결과는 무엇인지 | "입력 데이터는 어디서 오고, 결과는 어떤 형태여야 하나요?" | - |
   | 5 | **기존 코드 제약** | 수정해야 할 기존 코드, 건드리면 안 되는 영역 | "기존 코드 중 이 작업과 관련된 부분이 있나요?" | `[자동감지]` Step 0 |
   | 6 | **기술 스택/방식** | 사용할 기술, 라이브러리, 패턴에 대한 선호 | "특별히 사용하고 싶은 기술이나 패턴이 있나요?" | `[자동감지]` Step 0 |
   | 7 | **비기능 요구사항** | 성능, 보안, 접근성 등 품질 속성 | "성능, 보안, 에러 처리 등에서 특별히 신경 써야 할 부분이 있나요?" | - |
   | 8 | **범위 경계** | 이번에 포함하지 않을 것 (out of scope) | "이번 작업에서 의도적으로 제외할 것이 있나요?" | - |

   **자동감지 규칙**: `[자동감지]` 항목(5, 6)은 질문 스킵. `[감지실패]`면 직접 질문. Ideation 기록이 있으면 ideation 우선, 자동감지는 보충.

   ### 1b. 복잡도 판단

   인터뷰 중 파악된 정보를 바탕으로 복잡도를 판단하세요 (각 신호당 0 또는 1점):

   | 신호 | 1점 조건 |
   |------|----------|
   | **범위** | 5개 이상 파일 변경 예상, 또는 2개 이상 모듈에 걸침 |
   | **신규 엔티티** | 새 데이터 모델·API·모듈을 2개 이상 도입 |
   | **설계 대안** | 구현 방식이 2가지 이상, 각각 다른 트레이드오프 존재 |
   | **불확실성** | 기술 스택 미정, 외부 의존성 미검증, 성능/확장성 요구 |

   - **0~1점**: 필수 확인 항목 충족 후 Step 1c로 진행
   - **2점 이상**: 아래 **설계 탐색**을 추가로 진행하세요:
     1. 식별된 설계 대안을 2~3가지 옵션으로 정리하세요 (각 옵션에 장단점 명시)
     2. `AskUserQuestion`으로 옵션을 제시하고 사용자가 선택하도록 하세요
     3. 불확실성이 남아있다면 추가 질문으로 해소하세요
     4. 선택된 설계 방향을 Step 2의 입력으로 사용하세요

   ### 1c. 요구사항 내부 정리 (체크포인트 없음)

   인터뷰 완료 후, 수집한 내용을 내부적으로 정리하세요. **별도 확인을 받지 않고** 바로 Step 2(디자인/보안 자동감지)와 Step 3(스펙 작성)으로 진행합니다. 사용자 확인은 Step 3 이후의 **통합 체크포인트**에서 한 번에 받습니다.

   정리할 내용:
   ```
   - 목적, 대상 사용자, 핵심 유즈케이스, 입출력
   - 기존 코드 제약 [자동감지 또는 인터뷰]
   - 기술 스택 [자동감지 또는 인터뷰]
   - 비기능 요구사항, 범위 밖
   - 복잡도 점수
   ```

2. **디자인 리뷰 체크포인트** (UI/프론트엔드 관련 플랜인 경우)

   Step 0에서 탐색한 코드와 요구사항을 기반으로, 이 플랜이 UI/프론트엔드 관련인지 판단하세요.

   **UI 관련 판단 기준** -- 아래 중 하나라도 해당하면 UI 관련 플랜입니다:
   - 변경 대상 파일 확장자: `.tsx`, `.jsx`, `.vue`, `.svelte`, `.css`, `.scss`, `.html`
   - 요구사항 키워드: UI, 화면, 페이지, 컴포넌트, 레이아웃, 스타일, 디자인, 폼, 버튼, 모달, 대시보드, 반응형

   **UI 관련 플랜인 경우** (질문 없이 자동 적용):

   a. **DESIGN.md 존재 확인**: 프로젝트 루트에 DESIGN.md가 있는지 확인하세요.
      - **있는 경우**: DESIGN.md를 읽어 디자인 토큰을 파악하세요
      - **없는 경우**: `/vs-design-init` 안내를 통합 체크포인트의 부가 정보로 표시하세요

   b. **자동 포함**: 스펙에 "UI/UX Design" 섹션을 자동 포함하세요 (DESIGN.md 참조, 레이아웃, 반응형).
      태스크 분해 시 "디자인 시스템 검증" 태스크를 자동 추가하세요.

   c. **적용 근거 표시**: 통합 체크포인트에서 다음과 같이 표시:
      ```
      [자동 포함] UI/UX Design 섹션 — 근거: 변경 대상에 .tsx 파일 포함 (제거하려면 '스펙 수정' 선택)
      ```

   **UI 관련 플랜이 아닌 경우:** 이 단계를 건너뛰고 Step 2a로 진행하세요.

2a. **보안 키워드 감지** (자동, 질문 없음)

   요구사항 인터뷰(Step 1)와 코드베이스 탐색(Step 0) 결과에서 보안 키워드를 자동으로 감지합니다.

   **보안 키워드 목록:**
   - `auth`, `login`, `password`, `payment`, `token`, `session`, `encrypt`, `API key`
   - `oauth`, `jwt`, `credential`, `secret`, `certificate`, `permission`, `role`

   **감지 방법:**
   - 사용자의 요구사항 설명에서 위 키워드 포함 여부를 확인하세요
   - Step 0에서 탐색한 관련 파일명/코드에서 키워드를 검색하세요

   **보안 키워드가 감지된 경우** (질문 없이 자동 포함):
   - 스펙에 **Security Considerations** 섹션을 자동 포함하세요
   - 적용 근거를 통합 체크포인트에서 표시:
     ```
     [자동 포함] Security Considerations 섹션 — 근거: 키워드 감지 ({detected keywords}) (제거하려면 '스펙 수정' 선택)
     ```

   **보안 키워드가 감지되지 않은 경우:** 이 단계를 건너뛰고 Step 3으로 진행하세요.

3. **SDD 스펙 작성**
   다음 구조로 스펙을 작성하세요:

   ```
   ## Overview
   한 문단 요약 — 무엇을, 누구를 위해, 왜

   ## Requirements
   - 기능 요구사항 (MUST / SHOULD / COULD)
   - 비기능 요구사항 (성능, 보안, 접근성)

   ## Data Model
   - 엔티티 정의, 필드, 타입, 관계

   ## API / Interface
   - 함수 시그니처, 파라미터, 리턴 타입
   - 에러 케이스

   ## Edge Cases
   - 알려진 엣지 케이스와 처리 방법 (최소 3개 이상)

   ## Success Criteria
   - 이 기능이 성공했다고 판단할 수 있는 구체적 지표
   - 정량적 기준 (응답시간, 에러율, 커버리지 등)
   - 정성적 기준 (사용자 경험, 코드 품질 등)

   ## Security Considerations (Step 2a에서 보안 키워드 감지 + 사용자 승인 시 포함)
   - Authentication / Authorization
   - Data Protection
   - Input Validation
   - Session Management
   ```

   **스펙 품질 체크리스트** — 작성 후 아래 항목을 자가 검증하세요:
   - [ ] Overview가 what/whom/why를 모두 커버하는가
   - [ ] MUST 요구사항이 테스트 가능한 형태로 기술되었는가
   - [ ] Data Model에 타입과 제약조건이 명시되었는가
   - [ ] API에 에러 케이스가 포함되었는가
   - [ ] Edge Cases가 3개 이상 식별되었는가
   - [ ] Success Criteria를 테스트로 변환할 수 있는가

   스펙 작성 후 **바로 플랜 생성(Step 4)과 태스크 분해(Step 5)까지 진행**하세요. 사용자 확인은 태스크 분해 후 통합 체크포인트(Step 6)에서 한 번에 받습니다.

4. **플랜 생성**
   - 스펙으로 Bash 도구로 `vs --json plan create --title "기능명" --spec "작성한 스펙 전문"` 명령을 실행하세요
   - 스펙이 길면 heredoc을 사용하세요:
     ```bash
     vs --json plan create --title "기능명" --spec "$(cat <<'SPEC'
     스펙 내용...
     SPEC
     )"
     ```

5. **태스크 분해**
   스펙을 태스크로 분해하세요. 각 태스크는:
   - 한 세션(15~30분)에 완료 가능한 크기
   - 명확한 spec (구현 상세)
   - 검증 가능한 acceptance criteria (**AC 네이밍 컨벤션** 참조)
   - 의존성 순서대로 정렬 (Data Model → Core Logic → API → Integration)
   - 각 태스크에 S/M/L **복잡도 라벨**을 표시 (S: 15분 이내, M: 15~30분, L: 30분 초과 — 분할 검토)

   **AC 네이밍 컨벤션** (권장):
   acceptance criteria는 `AC01`, `AC02`, ... 형식으로 번호를 매기세요.
   이렇게 하면 tdd-implementer가 테스트 이름에 AC 번호를 포함하고,
   verifier가 테스트-AC 매핑을 자동으로 검증할 수 있습니다.

   ```
   # acceptance 작성 예시
   acceptance: "AC01: 사용자가 이메일로 로그인할 수 있어야 한다.
                AC02: 비밀번호 5회 실패 시 계정이 잠겨야 한다.
                AC03: 로그인 성공 시 JWT 토큰을 반환해야 한다."

   # 대응하는 테스트 이름
   it('AC01: 이메일 로그인 성공')
   it('AC02: 비밀번호 5회 실패 시 계정 잠금')
   it('AC03: 로그인 성공 시 JWT 반환')
   ```
   - **Scope 규칙** (권장): 각 태스크에 수정 범위를 명시하세요:
     - `allowed_files` — 해당 태스크에서 수정할 파일 목록. Step 2에서 파악한 관련 파일과 스펙의 API/Data Model에서 도출
     - `forbidden_patterns` — 해당 태스크와 무관한 영역. 보통 core/공통/설정 모듈, 다른 태스크가 담당하는 파일
     - 이 정보는 에이전트의 범위 이탈을 방지하는 안전장치입니다

     **예시:**
     ```
     allowed_files: src/api/auth.ts, src/models/user.ts, tests/auth.test.ts
     forbidden_patterns: src/core/*, src/config/*, migrations/*
     ```

   **보안 감사 태스크 추가** (Step 2a에서 보안 키워드가 감지된 경우):
   - 태스크 분해 시 마지막에 "보안 감사" 태스크를 추가하는 것을 제안하세요
   - 이 태스크는 `/vs-security` 스킬을 실행하여 구현된 코드의 보안 취약점을 점검합니다
   - 보안 감사 태스크 예시:
     ```
     title: "보안 감사 — OWASP Top 10 점검"
     spec: "vs-security 스킬을 사용하여 변경된 파일에 대해 OWASP Top 10 보안 점검을 수행합니다. A01(접근 제어), A02(암호화), A03(인젝션), A07(인증), A09(보안 로깅) 카테고리를 검사합니다."
     acceptance: "critical/high 심각도 취약점이 0건이어야 합니다. medium 이하는 리포트에 기록합니다."
     depends_on: [모든 구현 태스크]
     ```

5a. **비판적 검토 자동 디스패치** (조건부)

   태스크 분해 완료 후, 스펙 품질을 에이전트로 자동 검토합니다.

   **디스패치 조건**: 아래 중 하나라도 충족하면 디스패치합니다:
   - 분해된 태스크 수 >= 4
   - 복잡도 점수(Step 1b) >= 2

   조건 미충족 시 이 단계를 스킵하고 Step 6으로 바로 진행하세요.

   **디스패치 실행**:
   Agent 도구로 plan-critical-reviewer를 디스패치하세요 (model: sonnet):
   ```
   당신은 plan-critical-reviewer 에이전트입니다.
   agents/plan-critical-reviewer.md의 Execution Process를 따라 실행하세요.

   plan_id: {plan_id}
   plan_spec: {생성된 스펙 전문}
   complexity_score: {Step 1b에서 계산된 복잡도 점수}
   ```

   UI 관련 플랜인 경우 Step 7a의 plan-design-reviewer와 **동시에 병렬 디스패치**할 수 있습니다 (각 에이전트의 관심사가 다르므로 독립 반영).

   **결과 반영**:
   에이전트가 정상 응답하면:
   1. 결과의 `summary` 배열을 순회하세요
   2. 각 항목의 `dimension`/`action`/`detail`을 기반으로 스펙의 해당 섹션(Requirements/Edge Cases/Success Criteria)에 직접 추가/수정하세요
   3. 반영 내역을 내부적으로 보관하세요 (Step 6 통합 체크포인트에서 "자동 검토 반영 사항" 섹션으로 표시)

   **폴백**:
   에이전트가 60초 내 응답하지 않거나 비정상 JSON을 반환하면:
   - "에이전트 검토 실패 — 인라인 검토로 대체합니다" 메시지를 표시하세요
   - 기존 Step 6a의 인라인 검토 로직(6가지 관점)을 직접 실행하세요

6. **플랜 리뷰**
   태스크 분해 결과를 아래 관점으로 직접 비판적 검토하세요:
   - **태스크 누락**: 스펙의 모든 요구사항이 태스크로 커버되는가?
   - **의존성 순서**: 태스크 간 선후관계가 올바른가? 순환 의존은 없는가?
   - **성공 지표**: 스펙의 Success Criteria가 구체적이고 측정 가능한가?
   - **플랜 크기**: 태스크가 6~7개이면 플랜 분할을 검토하고, 8개 이상이면 반드시 분할하세요

   Step 6의 검토 결과를 스펙/태스크에 직접 반영하세요.

   ### 통합 체크포인트 (유일한 사용자 확인)

   스펙 + 태스크 분해 결과를 **한 화면에** 보여주고 `AskUserQuestion`으로 1회 확인받으세요:

   ```
   ## 요구사항 요약
   - **목적**: ...
   - **기존 코드 제약**: ... [자동감지]
   - **기술 스택**: ... [자동감지]
   - (나머지 항목)
   {자동 포함 섹션이 있으면}
   - [자동 포함] UI/UX Design — 근거: {1줄} (제거하려면 '스펙 수정' 선택)
   - [자동 포함] Security — 근거: {1줄} (제거하려면 '스펙 수정' 선택)

   ## 스펙 요약
   (Overview, Requirements MUST/SHOULD/COULD, Edge Cases, Success Criteria 핵심 내용)

   ## 태스크 분해 ({N}개)
   | # | 태스크 | 복잡도 | 의존성 |
   |---|--------|--------|--------|
   | 1 | ... | S | - |
   | 2 | ... | M | T1 |

   {Step 5a에서 에이전트가 실행된 경우에만 표시}
   ## 자동 검토 반영 사항 ({N}건)
   - [{dimension}] {action}: {detail}
   (모든 항목 pass이면: "자동 검토 반영 사항: 없음 (모든 항목 통과)")
   (5개 이상 개선필요이면: "⚠ 다수 개선 사항이 반영되었습니다. 스펙을 주의 깊게 검토하세요.")
   {에이전트가 스킵된 경우: 이 섹션을 표시하지 않음}
   ```

   - question: "스펙과 태스크를 검토해주세요. 어떻게 진행할까요?"
   - header: "플랜 확인"
   - multiSelect: false
   - 선택지:
     - label: "승인", description: "현재 스펙과 태스크로 플랜을 확정합니다"
     - label: "스펙 수정", description: "스펙 내용을 수정합니다 (자동 포함 섹션 제거 포함)"
     - label: "태스크 수정", description: "태스크 분해를 조정합니다"
     - label: "재검토", description: "요구사항을 다시 비판적으로 검토합니다 (에이전트 재실행)"

   - "승인" → Step 7로 진행
   - "스펙 수정" → 피드백 반영 후 통합 체크포인트 재표시
   - "태스크 수정" → 피드백 반영 후 통합 체크포인트 재표시
   - "재검토" → Step 5a의 에이전트 디스패치를 재실행합니다 (조건 무시, 무조건 실행). 결과를 다시 반영하고 통합 체크포인트를 재표시하세요.

   ### 6a. 비판적 검토 (폴백 또는 재검토 시)

   > Step 5a의 에이전트가 정상 실행된 경우, 이 로직은 사용자가 "재검토"를 선택했을 때만 실행됩니다.
   > Step 5a의 에이전트가 실패/타임아웃된 경우, 이 로직이 폴백으로 자동 실행됩니다.

   아래 6가지 관점으로 요구사항을 검토하세요:
   | # | 관점 | 판정 기준 |
   |---|------|-----------|
   | 1 | 모호성 | MUST 요구사항이 테스트 가능한 형태가 아니면 "개선필요" |
   | 2 | 엣지케이스 누락 | 실패 시나리오가 3개 미만이면 "개선필요" |
   | 3 | 기술 실현성 | 기술 스택/방식이 미정이거나 검증 안됐으면 "확인필요" |
   | 4 | 요구사항 충돌 | 2개 이상 요구사항이 상호 모순이면 "개선필요" |
   | 5 | 범위 적절성 | out of scope가 정의되지 않았으면 "확인필요" |
   | 6 | 비기능 요구사항 | 성능/보안/에러 처리 중 언급 없는 것이 있으면 "확인필요" |

   검토 결과를 **자동으로 전체 반영**하고 결과만 인라인 표시합니다 (별도 선택/루프 없음).
   반영 후 통합 체크포인트로 복귀하세요.

7. **태스크 생성**
   - 승인된 각 태스크를 Bash 도구로 `vs --json task create --plan <id> --title "..." --spec "..." --acceptance "..."` 명령을 실행하여 생성하세요
   - 복합 태스크는 `--parent <id>` 옵션을 사용하여 서브태스크로 구성하세요
   - 의존성이 있는 태스크는 `--depends-on "id1,id2"` 옵션을 추가하세요
   - scope 규칙이 정의된 태스크는 `--allowed-files "파일1,파일2"` 및 `--forbidden-patterns "패턴1,패턴2"` 옵션을 추가하세요
   - spec과 acceptance를 반드시 포함하세요

7a. **디자인 점수 검증 & 후속 조치** (UI 관련 플랜인 경우)

   Step 2에서 UI 관련 플랜으로 판단된 경우에만 실행합니다.

   a. **plan-design-reviewer 에이전트 디스패치**:
      - Agent 도구로 plan-design-reviewer를 디스패치하세요 (model: sonnet):

        ```
        agents/plan-design-reviewer.md의 Execution Process를 따라 실행하세요.

        plan_spec: {생성된 스펙 전문}
        design_md: {DESIGN.md 내용 또는 null}
        ```
      - 결과의 design_score (A~F 등급)와 critical_gaps를 수집하세요

   b. **점수별 후속 조치**:
      - **A, B 등급** (7.0 이상): 결과를 요약 표시하고 Step 7b로 진행
        ```
        ✓ Design Score: {등급} ({점수}/10) — {critical_gaps 수}개 개선 제안
        ```
      - **C 등급** (5.0~6.9): 결과를 표시하고 `AskUserQuestion`으로 선택:
        - question: "Design Score가 {등급}입니다. {critical_gaps 요약}. 스펙을 보완할까요?"
        - header: "디자인 리뷰 결과"
        - 선택지:
          - label: "스펙 보완 (권장)", description: "critical_gaps를 반영하여 스펙의 UI/UX 섹션을 보완합니다"
          - label: "그대로 진행", description: "현재 스펙으로 진행하고 구현 시 보완합니다"
        - "스펙 보완" 선택 시: critical_gaps의 suggestions를 스펙에 반영하고 관련 태스크 AC를 보강하세요
      - **D, F 등급** (5.0 미만): 결과를 경고와 함께 표시하고 `AskUserQuestion`으로 선택:
        - question: "Design Score가 {등급}으로 낮습니다. {critical_gaps 상세}. 스펙을 대폭 수정해야 합니다."
        - header: "디자인 리뷰 경고"
        - 선택지:
          - label: "스펙 재작성 (강력 권장)", description: "UI/UX 섹션을 처음부터 다시 작성하고 태스크를 재분해합니다"
          - label: "부분 보완", description: "critical_gaps만 반영합니다"
          - label: "무시하고 진행", description: "리스크를 감수하고 현재 스펙으로 진행합니다"
        - "스펙 재작성" 선택 시: UI/UX 섹션을 critical_gaps + suggestions 기반으로 전면 재작성하고 관련 태스크를 재분해하세요
        - "부분 보완" 선택 시: critical_gaps의 suggestions를 스펙에 반영하세요
      - **N/A** (UI 요소 미식별): 이 단계를 조용히 스킵하세요

8. **워크트리 추천**

   태스크 생성이 완료되면 작업 환경의 격리 필요성을 판단하세요.

   **추천 조건** — 아래 중 하나라도 해당하면 워크트리를 추천하세요:
   - 복잡도 점수 >= 2점 (Step 1b에서 계산)
   - 생성된 태스크 수 ≥ 4개
   - `allowed_files`에서 파악한 변경 대상 파일이 3개 이상 디렉토리/모듈에 분포
   - 백로그에서 승격된 플랜이고 원본 항목의 `complexity_hint`가 `complex`인 경우
     (Bash 도구로 `vs --json backlog list --status planned`를 실행하여 해당 플랜과 연결된 백로그 항목을 확인)

   **추천하지 않는 경우:**
   - 이미 워크트리 안에 있을 때 (현재 경로에 `/worktrees/`가 포함)
   - 태스크가 2개 이하이고 단일 모듈 내 변경

   해당 조건을 충족하면 `AskUserQuestion`으로 선택지를 제시하세요:
   - header: "워크트리 추천"
   - question: "이 플랜은 {추천 근거}. 격리된 워크트리에서 작업하시겠습니까?"
   - multiSelect: false
   - 선택지:
     - label: "워크트리에서 작업", description: "/vs-worktree로 격리된 환경을 생성합니다"
     - label: "현재 브랜치에서 작업", description: "별도 격리 없이 바로 작업을 시작합니다"
     - label: "나중에 결정", description: "지금은 건너뛰고 나중에 /vs-worktree를 실행합니다"

   - "워크트리에서 작업" 선택 시: `/vs-worktree`를 안내하세요
   - "현재 브랜치에서 작업" 선택 시: `/vs-next`를 안내하세요
   - "나중에 결정" 선택 시: Step 9로 진행하세요
