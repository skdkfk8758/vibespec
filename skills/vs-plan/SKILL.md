# Spec → Plan 생성 워크플로우

사용자의 요구사항을 SDD 스펙으로 정리하고 실행 가능한 플랜으로 분해합니다.

## Steps

1. **요구사항 수집**
   - 사용자에게 무엇을 만들고 싶은지 물어보세요
   - 요구사항의 복잡도를 판단하세요
   - 복잡한 기능 (설계 선택지가 여러 개, 트레이드오프 존재):
     → `brainstorming` 스킬을 실행하여 설계 옵션을 탐색하세요
     → brainstorming 결과(선택된 옵션 + 설계 스펙)를 다음 단계의 입력으로 사용하세요
   - 단순한 기능 (명확한 구현 방향):
     → 직접 핵심 요구사항, 제약조건, 기술 스택을 파악하세요
     → 애매한 부분은 구체적으로 질문하여 명확히 하세요

2. **SDD 스펙 작성**
   다음 구조로 스펙을 작성하세요:

   ```
   ## Overview
   한 문단 요약 — 무엇을, 누구를 위해, 왜

   ## Requirements
   - 기능 요구사항 (MUST / SHOULD / COULD)
   - 비기능 요구사항 (성능, 보안, 접근성)

   ## Data Model
   - 엔티티 정의, 필드, 타입, 관계

   ## API / Interface
   - 함수 시그니처, 파라미터, 리턴 타입
   - 에러 케이스

   ## Edge Cases
   - 알려진 엣지 케이스와 처리 방법

   ## Success Criteria
   - 이 기능이 성공했다고 판단할 수 있는 구체적 지표
   - 정량적 기준 (응답시간, 에러율, 커버리지 등)
   - 정성적 기준 (사용자 경험, 코드 품질 등)
   ```

   스펙을 사용자에게 보여주고 승인을 받으세요.

3. **플랜 생성**
   - 승인된 스펙으로 `vp_plan_create`를 호출하세요
   - title: 기능명, spec: 작성한 스펙 전문

4. **태스크 분해**
   스펙을 태스크로 분해하세요. 각 태스크는:
   - 한 세션(15~30분)에 완료 가능한 크기
   - 명확한 spec (구현 상세)
   - 검증 가능한 acceptance criteria
   - 의존성 순서대로 정렬

5. **플랜 리뷰**
   `plan-reviewer` 에이전트를 실행하여 태스크 분해 결과를 비판적으로 검토하세요.
   리뷰 관점:
   - **태스크 누락**: 스펙의 모든 요구사항이 태스크로 커버되는가?
   - **의존성 순서**: 태스크 간 선후관계가 올바른가? 순환 의존은 없는가?
   - **성공 지표**: 스펙의 Success Criteria가 구체적이고 측정 가능한가?
   - **플랜 크기**: 태스크가 5개를 초과하면 플랜 분할을 제안하세요

   리뷰 결과를 사용자에게 보여주고, 반영 여부를 결정받으세요.
   개선점을 반영했으면 사용자에게 최종 태스크 목록의 승인을 받으세요.

6. **태스크 생성**
   - 승인된 각 태스크를 `vp_task_create`로 생성하세요
   - 복합 태스크는 parent_id를 사용하여 서브태스크로 구성하세요
   - spec과 acceptance를 반드시 포함하세요

7. **결과 확인**
   - `vp_plan_get`으로 생성된 플랜과 태스크 트리를 보여주세요
   - `vp_context_save`로 플랜 생성 내용을 저장하세요
   - `/vs-next`로 바로 작업을 시작할 수 있다고 안내하세요
