# Plan Review Agents

## 비판적 검토 자동 디스패치 (Step 5a)

태스크 분해 완료 후, 스펙 품질을 에이전트로 자동 검토합니다.

**디스패치 조건**: 아래 중 하나라도 충족하면 디스패치합니다:
- 분해된 태스크 수 >= 4
- 복잡도 점수(Step 1b) >= 2

조건 미충족 시 이 단계를 스킵하고 Step 6으로 바로 진행하세요.

**디스패치 실행**:
Agent 도구로 plan-critical-reviewer를 디스패치하세요 (model: sonnet):
```
당신은 plan-critical-reviewer 에이전트입니다.
agents/plan-critical-reviewer.md의 Execution Process를 따라 실행하세요.

plan_id: {plan_id}
plan_spec: {생성된 스펙 전문}
complexity_score: {Step 1b에서 계산된 복잡도 점수}
```

UI 관련 플랜인 경우 Step 7a의 plan-design-reviewer와 **동시에 병렬 디스패치**할 수 있습니다 (각 에이전트의 관심사가 다르므로 독립 반영).

**결과 반영**:
에이전트가 정상 응답하면:
1. 결과의 `summary` 배열을 순회하세요
2. 각 항목의 `dimension`/`action`/`detail`을 기반으로 스펙의 해당 섹션(Requirements/Edge Cases/Success Criteria)에 직접 추가/수정하세요
3. 반영 내역을 내부적으로 보관하세요 (Step 6 통합 체크포인트에서 "자동 검토 반영 사항" 섹션으로 표시)

**폴백**:
에이전트가 60초 내 응답하지 않거나 비정상 JSON을 반환하면:
- "에이전트 검토 실패 — 인라인 검토로 대체합니다" 메시지를 표시하세요
- 아래 "인라인 비판적 검토"를 직접 실행하세요

## 인라인 비판적 검토 (Step 6a — 폴백 또는 재검토 시)

> Step 5a의 에이전트가 정상 실행된 경우, 이 로직은 사용자가 "재검토"를 선택했을 때만 실행됩니다.
> Step 5a의 에이전트가 실패/타임아웃된 경우, 이 로직이 폴백으로 자동 실행됩니다.

아래 6가지 관점으로 요구사항을 검토하세요:
| # | 관점 | 판정 기준 |
|---|------|-----------|
| 1 | 모호성 | MUST 요구사항이 테스트 가능한 형태가 아니면 "개선필요" |
| 2 | 엣지케이스 누락 | 실패 시나리오가 3개 미만이면 "개선필요" |
| 3 | 기술 실현성 | 기술 스택/방식이 미정이거나 검증 안됐으면 "확인필요" |
| 4 | 요구사항 충돌 | 2개 이상 요구사항이 상호 모순이면 "개선필요" |
| 5 | 범위 적절성 | out of scope가 정의되지 않았으면 "확인필요" |
| 6 | 비기능 요구사항 | 성능/보안/에러 처리 중 언급 없는 것이 있으면 "확인필요" |

검토 결과를 **자동으로 전체 반영**하고 결과만 인라인 표시합니다 (별도 선택/루프 없음).
반영 후 통합 체크포인트로 복귀하세요.
