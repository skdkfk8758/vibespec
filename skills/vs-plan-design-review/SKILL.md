# Plan Design Review (플랜 디자인 리뷰)

플랜 단계에서 UI/UX 설계를 7개 차원으로 검증합니다. 코드를 작성하기 전에 디자인 결정의 누락과 약점을 잡습니다.

## When to Use

**사용하세요:**
- `/vs-plan` 완료 후, 구현 시작 전 — UI/프론트엔드 관련 플랜일 때
- 디자인 품질을 체계적으로 검증하고 싶을 때
- vs-plan Step 3에서 자동 제안될 때

**사용하지 마세요:**
- 구현 완료 후 시각 감사 → `/vs-design-review` (80항목 감사)
- 디자인 시스템 생성 → `/vs-design-init`
- 백엔드 전용 플랜

## Prerequisites

- 활성 플랜이 존재해야 합니다
- 플랜 스펙에 UI/프론트엔드 관련 요구사항이 포함되어야 합니다

## Steps

### Step 1: 플랜 스펙 로드 및 UI 요소 식별

1. Bash 도구로 `vs --json plan list --status active` 실행하여 활성 플랜 조회
2. 플랜이 여러 개면 `AskUserQuestion`으로 선택
3. `vs --json plan show <plan_id>` 실행하여 스펙 로드
4. 스펙에서 UI 관련 요소를 식별:
   - 화면/페이지/뷰 목록
   - 컴포넌트 목록
   - 사용자 인터랙션 (폼, 버튼, 네비게이션)
   - 데이터 표시 영역
5. UI 요소가 없으면: "이 플랜에 UI 요소가 식별되지 않았습니다." 안내 후 종료
6. DESIGN.md 존재 여부 확인 — 있으면 토큰 기반 검증, 없으면 일반 디자인 원칙 적용

### Step 2: 7패스 디자인 리뷰

아래 7개 차원을 순서대로 검증합니다. 각 차원에 대해:
- **0-10 점수** 부여
- **10의 모습** 설명 (이 차원에서 완벽한 플랜은 어떤 형태인지)
- 점수가 **7 미만**이면 구체적 수정 제안 + `AskUserQuestion`으로 승인
- 점수가 **7 이상**이면 요약만 출력하고 다음 패스로

---

#### Pass 1: Information Architecture (정보 구조) {0-10}

**검증 항목:**
- 각 화면의 콘텐츠 계층 (primary / secondary / tertiary)이 정의되었는가
- 네비게이션 구조와 페이지 간 관계가 명확한가
- 정보 그룹핑과 순서가 사용자 멘탈 모델에 맞는가

**10의 모습:** 모든 화면에 콘텐츠 우선순위가 정의되고, 네비게이션 트리가 3단계 이내이며, 관련 정보가 논리적으로 그룹핑됨.

---

#### Pass 2: Interaction State Coverage (상태 커버리지) {0-10}

**검증 항목:**
- 각 UI 요소에 대해 5가지 상태가 정의되었는가:
  - **Empty** — 데이터 없음 (첫 사용, 검색 결과 0건)
  - **Loading** — 데이터 로딩 중 (스켈레톤, 스피너)
  - **Error** — 실패 상태 (네트워크 에러, 권한 없음, 404)
  - **Success** — 정상 상태 (데이터 표시)
  - **Partial** — 부분 로드, 부분 실패, 오프라인
- "No items found." 같은 제네릭 빈 상태가 아닌 맥락에 맞는 가이드가 있는가

**10의 모습:** 모든 데이터 표시 영역에 5가지 상태가 명시되고, 각 상태에 구체적인 UI 동작과 메시지가 정의됨.

---

#### Pass 3: User Journey (사용자 여정) {0-10}

**검증 항목:**
- 진입 → 핵심 액션 → 완료까지의 전체 플로우가 정의되었는가
- 각 단계에서 사용자가 취할 수 있는 행동이 명확한가
- 이탈 지점과 복귀 경로가 고려되었는가
- 첫 사용자(onboarding)와 재방문 사용자의 경험 차이가 있는가

**10의 모습:** 메인 플로우 + 2개 이상 대안 플로우가 정의되고, 각 분기점에서의 사용자 행동이 명시됨.

---

#### Pass 4: AI Slop Risk (AI 생성 코드 위험) {0-10}

**검증 항목:**
- "카드형 레이아웃 + 그라디언트 배경" 같은 AI 디폴트 패턴에 빠지지 않는가
- 디자인이 프로젝트의 정체성을 반영하는가 (DESIGN.md 토큰 참조)
- 장식적 요소(아이콘, 그림자, 보더)가 의미를 전달하는가
- 색상/간격/타이포가 일관된 시스템을 따르는가 (임의 값 아닌)

**10의 모습:** DESIGN.md 토큰만 사용, 장식 요소에 목적이 명시됨, 프로젝트 고유 스타일이 반영됨.

---

#### Pass 5: Design System Alignment (디자인 시스템 정합성) {0-10}

**검증 항목:**
- DESIGN.md의 색상/타이포/간격 토큰이 스펙에 참조되었는가
- 기존 컴포넌트 재사용이 명시되었는가 (신규 컴포넌트 최소화)
- 새 컴포넌트가 필요한 경우 디자인 시스템과의 관계가 정의되었는가

**10의 모습:** 스펙의 모든 UI 요소가 기존 컴포넌트 또는 DESIGN.md 토큰으로 표현됨. 신규 컴포넌트에 토큰 매핑이 정의됨.

**DESIGN.md 없는 경우:** 이 패스를 SKIP하고, vs-design-init 실행을 제안.

---

#### Pass 6: Responsive & Accessibility (반응형/접근성) {0-10}

**검증 항목:**
- 모바일/태블릿/데스크톱 레이아웃 동작이 정의되었는가
- 터치 타겟 크기(44px+)가 고려되었는가
- 키보드 네비게이션이 가능한가
- 스크린리더 지원이 고려되었는가 (ARIA, 시맨틱 HTML)
- 색상 대비가 WCAG AA 이상인가

**10의 모습:** 3개 브레이크포인트 레이아웃이 명시되고, 키보드/스크린리더 동작이 정의되며, WCAG AA 준수.

---

#### Pass 7: Unresolved Design Decisions (미결 디자인 판단) {0-10}

**검증 항목:**
- "나중에 결정", "TBD", "TODO" 같은 미결 항목이 남아있는가
- 사용자에게 영향을 주는 디자인 판단이 개발자에게 위임되지 않았는가
- 복수 디자인 옵션이 미결인 채로 남아있는가

**10의 모습:** 모든 디자인 판단이 확정됨. 개발자가 UI 결정을 내릴 필요 없음.

---

### Step 3: 종합 평가

#### Design Score 산출

7개 차원 점수의 가중 평균으로 A~F 등급을 산출하세요:

| 차원 | 가중치 |
|------|--------|
| State Coverage | 20% |
| User Journey | 15% |
| AI Slop Risk | 15% |
| Information Architecture | 15% |
| Design System Alignment | 10% |
| Responsive/A11y | 15% |
| Unresolved Decisions | 10% |

| 등급 | 점수 범위 |
|------|-----------|
| A | 9.0 ~ 10.0 |
| B | 7.5 ~ 8.9 |
| C | 6.0 ~ 7.4 |
| D | 4.5 ~ 5.9 |
| E | 3.0 ~ 4.4 |
| F | 0.0 ~ 2.9 |

#### 결과 리포트

```
## Plan Design Review Report

**플랜**: {title}
**Design Score**: {등급} ({가중 평균 점수}/10)

### 차원별 점수
| 차원 | 점수 | 상태 |
|------|------|------|
| Information Architecture | {N}/10 | {PASS/NEEDS WORK} |
| State Coverage | {N}/10 | ... |
| User Journey | {N}/10 | ... |
| AI Slop Risk | {N}/10 | ... |
| Design System Alignment | {N}/10 | ... |
| Responsive/A11y | {N}/10 | ... |
| Unresolved Decisions | {N}/10 | ... |

### 수정 사항
- {패스 N에서 적용한 수정 목록}

### 다음 단계 권장
- {점수 기반 권장 사항}
```

### Step 4: 결과 저장

1. 리포트를 context_log에 저장: `vs context save --summary "[plan-design-review] {등급}: {플랜 제목} — {주요 개선점 요약}"`
2. `AskUserQuestion`으로 다음 단계 제시:
   - "구현 시작" → `/vs-next` 안내
   - "추가 수정" → 특정 차원 재검토
   - "리포트만 저장" → 종료

## Rules

- 비UI 플랜에서 실행하면 즉시 종료
- 점수 7 미만 차원은 반드시 수정 제안 + 사용자 승인
- 플랜 스펙을 직접 수정하지 않고, 수정 제안만 제시 (사용자 승인 후 적용)
- DESIGN.md가 없으면 Pass 5를 SKIP하고 vs-design-init 제안
- context_log 저장 시 반드시 `[plan-design-review]` 태그 포함
